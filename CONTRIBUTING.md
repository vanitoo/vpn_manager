# Contributing

Спасибо за интерес к Warp Private Network.

## Перед изменениями

1. Проверьте существующие Issues и Pull Requests.
2. Для крупной функции сначала создайте Issue с описанием задачи.
3. Не включайте в коммиты `.env`, токены, базы пользователей, логи с персональными данными и резервные копии.

## Локальная разработка

```bash
git clone https://github.com/vanitoo/vpn_manager.git
cd vpn_manager
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python -m app.main
```

Для Windows используйте:

```powershell
.\.venv\Scripts\Activate.ps1
```

## Правила кода

- Python-код должен быть совместим с используемой версией Python проекта.
- Сохраняйте асинхронную модель aiogram и aiohttp.
- Не размещайте бизнес-логику непосредственно в клавиатурах и форматировании сообщений.
- SQL-операции держите в слоях хранения, а не размножайте по handlers.
- Новые интеграции должны корректно работать в отключённом состоянии.
- Не логируйте секреты, полные платёжные данные и токены.
- Пользовательские тексты должны быть понятными и не раскрывать внутренние ошибки без необходимости.

## Структура изменений

Новая функция обычно требует:

- кода;
- конфигурации в `.env.example`, если появляются параметры;
- безопасных значений по умолчанию;
- обновления README или профильной документации;
- записи в `CHANGELOG.md` в секции `Unreleased`;
- теста или понятного сценария ручной проверки.

## Коммиты

Используйте короткие сообщения в повелительной форме:

```text
Add support ticket priorities
Fix Remnawave expiration parsing
Update payment provider diagnostics
```

Один коммит должен описывать одно логически завершённое изменение.

## Pull Request

В описании укажите:

- что изменено;
- зачем это нужно;
- как проверить;
- какие переменные конфигурации добавлены;
- есть ли миграции базы;
- известные ограничения.

Не смешивайте рефакторинг, новую функцию и массовое форматирование без необходимости. Человечество и так создало достаточно диффов, которые невозможно проверить.

## Issues

### Bug report

Приложите:

- версию или commit SHA;
- способ запуска;
- ожидаемое поведение;
- фактическое поведение;
- шаги воспроизведения;
- обезличенный фрагмент лога.

### Feature request

Опишите пользовательский сценарий, проблему и ожидаемый результат. Не ограничивайтесь названием конкретной библиотеки или кнопки.

## Безопасность

Уязвимости не следует публиковать в обычном Issue. Используйте порядок из [SECURITY.md](SECURITY.md).
