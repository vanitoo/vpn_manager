# KamenevaBook Bot v2.7

Telegram-бот магазина электронных книг: каталог, оплата, выдача файлов, админка, бэкапы, прокси и модульные платежи.

## Запуск

```bash
cp .env.example .env
python -m app.main
```

Windows:

```bat
run.bat
```

Docker:

```bash
docker compose up -d --build
```


## Telegram-меню и команды

В v2.7 бот сам регистрирует команды Telegram при старте. После переноса на новый токен/аккаунт BotFather обычно достаточно запустить бота — меню появится автоматически.

```env
AUTO_SETUP_BOT_MENU=true
```

Команды для пользователей:

```text
/start - Открыть магазин
/books - Книжная полка
/get - Мои покупки
/author - Об авторе
/support - Поддержка
/id - Мой Telegram ID
```

Для администраторов дополнительно регистрируется:

```text
/admin - Панель администратора
```

Если автоматическая настройка не нужна:

```env
AUTO_SETUP_BOT_MENU=false
```

## Платежи

В v2.6 платежные провайдеры стали безопасными для запуска:

- если включить ЮKassa, но не заполнить `YOOKASSA_SHOP_ID` или `YOOKASSA_SECRET_KEY`, бот не падает;
- не настроенный провайдер скрывается от покупателя;
- статус виден в админке: `/admin` → `💳 Платежные системы`;
- если ключи ЮKassa неверные и API вернет 401/Unauthorized, провайдер отключится до перезапуска, а причина попадет в лог и админку.

Бот может работать на polling. Для внешних платежей webhook не обязателен: бот создает платежную ссылку, пользователь оплачивает, возвращается в Telegram и нажимает `✅ Я оплатил`. После этого бот проверяет статус платежа через API платежной системы.

### Включить несколько способов оплаты

```env
PAYMENT_PROVIDERS=stars,yookassa
```

Доступные значения:

```env
stars,yookassa,lava,platega
```

Также поддерживаются флаги:

```env
ENABLE_STARS=true
YOOKASSA_ENABLED=true
LAVA_ENABLED=false
PLATEGA_ENABLED=false
```

Если `PAYMENT_PROVIDERS` задан, он главный. Если пустой — бот смотрит на флаги.

### Telegram Stars

```env
PAYMENT_PROVIDERS=stars
STARS_RUB_PER_STAR=1.70
```

Цена книги хранится в рублях, а Stars считаются по формуле:

```text
stars = ceil(price_rub / STARS_RUB_PER_STAR)
```

### ЮKassa без webhook

```env
PAYMENT_PROVIDERS=stars,yookassa
YOOKASSA_SHOP_ID=123456
YOOKASSA_SECRET_KEY=test_xxxxxxxxxxxxxxxxx
YOOKASSA_RETURN_URL=

# Чеки
RECEIPT_REQUIRE_CONTACT=true
RECEIPT_FALLBACK_EMAIL=orders@example.com
RECEIPT_SAVE_CONTACT=true
```

Логика:

1. Пользователь нажимает `Купить книгу`.
2. Выбирает `ЮKassa`.
3. Если включены чеки и у пользователя нет контакта, бот просит email или телефон.
4. Бот создает платеж в ЮKassa с блоком receipt.
5. Бот показывает кнопку `Перейти к оплате`.
6. После оплаты пользователь нажимает `Я оплатил`.
7. Бот проверяет статус платежа через API ЮKassa.
8. Если статус `succeeded`, бот записывает покупку и выдает книгу.

Для тестирования можно поставить `RECEIPT_REQUIRE_CONTACT=false`; тогда бот не спрашивает email/телефон и подставляет `RECEIPT_FALLBACK_EMAIL`.

### Lava / Platega

Модули уже вынесены отдельно:

```text
app/payments/lava.py
app/payments/platega.py
```

Пока это заглушки. Их можно включать в конфиг, но для реальной работы нужно добавить актуальные API-методы создания счета и проверки статуса.

## Версия проекта

Единый источник версии находится в файле:

```text
app/version.py
```

Там задаются:

```python
APP_NAME = "KamenevaBook Bot"
APP_VERSION = "2.9.0"
BUILD_DATE = "2026-06-26"
DB_SCHEMA_VERSION = 7
```

В админке доступен пункт:

```text
ℹ️ О системе
```

Он показывает версию бота, схему базы, количество книг, пользователей, покупок, активные платежки и путь к логам.

## Схема базы

`DB_SCHEMA_VERSION` — это версия структуры SQLite-базы: таблицы, колонки, индексы и служебные поля. Она не равна версии бота.

Например, если добавили колонку `is_new`, таблицу `receipt_contacts` или служебную таблицу `app_meta`, это изменение схемы базы. При таких изменениях номер схемы увеличивается, чтобы бот понимал, какая структура базы ему нужна.

В v2.9 используется:

```text
DB_SCHEMA_VERSION=7
```

Почему 7: это текущий внутренний номер после основных этапов развития базы: книги, покупки, внешние платежи, настройки, контакты для чеков и служебная таблица `app_meta`.

## v3.0 — обработка обложек

При добавлении или редактировании книги обложку можно отправлять как обычное фото Telegram или как документ JPG/PNG/WEBP.

Бот сам сохраняет результат в папку книги:

```text
books/<slug>/
  cover.jpg
  cover_small.jpg
```

По умолчанию:

```env
COVER_WIDTH=1200
COVER_HEIGHT=1800
COVER_QUALITY=90
GENERATE_COVER_THUMBNAIL=true
COVER_THUMB_WIDTH=400
COVER_THUMB_HEIGHT=600
COVER_MIN_WIDTH=600
COVER_MIN_HEIGHT=900
```

`cover.jpg` — основная обложка 1200×1800.  
`cover_small.jpg` — миниатюра 400×600 для будущих витрин/быстрой загрузки.

Если исходная картинка маленькая, бот не запрещает загрузку, но пишет предупреждение в лог.


## v3.1: ожидающие платежи

Для внешних платежей без webhook бот теперь сохраняет ожидающий платеж.
Если пользователь оплатил в ЮKassa и вернулся по ссылке вида `https://t.me/<bot>?start=book_<slug>`, бот сам проверит оплату.
Если оплата еще не подтвердилась, в карточке книги появится кнопка `✅ Проверить оплату`.

Настройка времени жизни ожидающего платежа:

```env
PENDING_PAYMENT_TTL_MINUTES=60
```

## v3.2: баннеры экранов

Баннеры лежат в:

```text
content/banners/start.jpg
content/banners/bookshelf.jpg
content/banners/author.jpg
content/banners/support.jpg
```

Их можно менять из админки: `/admin` → `🖼 Баннеры`.

Уведомления о покупках админам управляются настройкой:

```env
ADMIN_NOTIFY_PURCHASES=true
```

Если админов несколько в `ADMIN_IDS`, уведомление о продаже придет каждому. Это штатное поведение.
